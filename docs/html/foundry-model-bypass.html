Using Foundry Models While Bypassing Copilot Studio's Built-in 4o

How to use GPT-5 (or any Azure AI Foundry model) for the heavy lifting while keeping Copilot Studio as the delivery channel — without its orchestrator model rewriting your answers.

1. The Problem

Copilot Studio in GCC only has GPT-4o. When you use generative orchestration or generative answers, 4o sits between your backend response and the user. It can paraphrase, truncate, add hedging, or drop citations — degrading a better answer from a more capable model.

What 4o can do to your response

  • Paraphrase or summarize a nuanced synthesis
  • Drop direct quotes, citations, or source links
  • Flatten structured output (change lists, confidence levels)
  • Add unnecessary caveats or hedging language
  • Truncate long responses

What we want instead

  • Foundry model (GPT-5, o3, etc.) generates the final answer
  • Answer reaches the user verbatim
  • Copilot Studio handles the channel (Teams, web) — not the thinking
  • Citations, quotes, and structure preserved exactly
  • Topic routing still works for multi-capability agent

2. The Architecture: Copilot Studio as Thin Shell

The key insight: Copilot Studio doesn't have to think. If you use classic topic authoring (not generative orchestration), you control exactly what gets sent and displayed. The built-in 4o model is only invoked when you use generative features.

User (Teams / Web) ↓ Copilot Studio ← Classic topics only, NO generative orchestration ↓ trigger phrases route to the right topic Topic: "News Clips" Topic: "Remarks Search" Topic: "Proofread" ↓ ↓ ↓ Custom Connector (APIM) Custom Connector (APIM) Custom Connector (APIM) ↓ ↓ ↓ Azure Function Azure Function Azure Function ↓ ↓ ↓ AI Foundry (GPT-5) AI Foundry (GPT-5) AI Foundry (GPT-5) ↓ ↓ ↓ Response returned verbatim via Message node — 4o never touches it

3. How to Keep 4o Out of the Picture

1
Turn off generative orchestration. In Copilot Studio → Settings → Generative AI, disable "AI-powered copilot." This stops 4o from interpreting user messages and choosing topics on its own. You'll use classic trigger phrases instead.
2
Use classic topic authoring. Each capability (clips, remarks, proofread) gets its own topic with trigger phrases. The topic calls the custom connector, stores the response in a variable, and displays it in a Message node. No "generative answers" node anywhere in the flow.
3
Display the response variable directly. In the Message node, insert the connector's output variable. Copilot Studio renders it as-is — no 4o involvement. For plain text, use {Topic.ResponseText}. For rich content, use an Adaptive Card.
4
Use Adaptive Cards for structured output. Return JSON from your backend, bind it to an Adaptive Card template in the Message node. Card content is rendered verbatim — 4o cannot rewrite Adaptive Card payloads. This is the strongest guarantee of fidelity.
5
Move the AI model call to Azure AI Foundry. Instead of calling Azure OpenAI directly from your Function, call an AI Foundry deployment. The Function code changes are minimal — swap the endpoint and deployment name. APIM still sits in front.

5. Code Change: Swap the Model Endpoint

The current src/shared/openai-client.ts creates an OpenAI client pointed at your Azure OpenAI resource. To use a Foundry deployment, you change the endpoint and deployment name. The openai npm package works with both.

// Today (Azure OpenAI) const client = new AzureOpenAI({ endpoint: process.env.AZURE_OPENAI_ENDPOINT, // https://<resource>.openai.azure.com apiVersion: "2024-10-21", azureADTokenProvider: getBearerTokenProvider(credential, scope), }); // After (AI Foundry) const client = new AzureOpenAI({ endpoint: process.env.AI_FOUNDRY_ENDPOINT, // https://<project>.services.ai.azure.com apiVersion: "2025-01-01", azureADTokenProvider: getBearerTokenProvider(credential, foundryScope), }); // Deployment name changes from "gpt-4o" to your Foundry deployment (e.g., "gpt-5")

Everything downstream — AI Search, Cosmos DB, the RAG pipeline — stays the same. The model swap is isolated to the client configuration.

6. Adaptive Card Strategy (Strongest Guarantee)

For maximum fidelity, return structured JSON from the backend and render it as an Adaptive Card. Copilot Studio passes the card through untouched.

Backend returns JSON

{ "summary": "Governor Stein has addressed...", "quotes": [ { "text": "We must invest in...", "source": "State of the State", "date": "2025-03-12" } ], "sourceCount": 4 }

Adaptive Card renders it

  • Summary text block — verbatim from GPT-5
  • Quote blocks with source attribution
  • Source count badge
  • No 4o rewriting — card payloads are opaque to the orchestrator

7. What You Lose (and Why It's Worth It)

You lose

  • Generative topic routing — 4o won't auto-detect intent; you need trigger phrases
  • Conversational follow-ups — 4o won't handle "tell me more" without explicit topic design
  • Fallback answers — no generative fallback for off-topic questions

You gain

  • Model freedom — use any Foundry model (GPT-5, o3, etc.) as it becomes available
  • Response fidelity — answers reach users exactly as the model wrote them
  • Predictability — no surprise paraphrasing or truncation from the orchestrator
  • Cost control — 4o token usage drops to near zero
Hybrid option: You can keep generative orchestration ON for topic routing (so users don't need exact trigger phrases) but still use Message nodes with direct variable output for responses. This lets 4o pick the right topic but keeps it out of the answer. Test whether 4o's topic selection adds enough value to justify the risk of it occasionally touching the response.

8. GCC Considerations

GCC / GCC-High constraints:
  • AI Foundry in GCC: Verify your Foundry project and model deployment are in a GCC-eligible region. As of early 2026, AI Foundry is available in GCC but model availability lags commercial. Check the Azure Government services-by-region page for current GPT-5 availability.
  • Copilot Studio in GCC: Classic topic authoring and custom connectors work in GCC. Generative orchestration availability may differ — but since we're turning it off, this doesn't matter.
  • Managed identity: DefaultAzureCredential works the same in GCC. The token audience for Foundry may use .us domain — verify against Foundry docs.
  • Data residency: All data stays in-tenant. Foundry deployments in GCC regions keep data within the government boundary.

9. Migration Steps (Checklist)

1
Create an AI Foundry project in your Azure subscription. Deploy the target model (GPT-5 or equivalent).
2
Grant the Function App's managed identity the appropriate Foundry RBAC role on the project.
3
Add AI_FOUNDRY_ENDPOINT and AI_FOUNDRY_DEPLOYMENT environment variables to the Function App.
4
Update openai-client.ts to use the Foundry endpoint for chat completions. Keep the Azure OpenAI endpoint for embeddings.
5
Test each capability end-to-end: clips query, remarks query, proofread. Verify response quality and latency.
6
In Copilot Studio, convert each topic to use Message nodes with direct variable output (no generative answers node).
7
(Optional) Build Adaptive Card templates for each capability's response format.
8
(Optional) Disable generative orchestration if you want full control over topic routing.
